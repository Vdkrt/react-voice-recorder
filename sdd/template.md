# [Название функции/системы] - SDD

## 1. Цели и нецели (Objectives & Non-Goals)
* **Цель:** Четкая бизнес-задача или поведение, которое мы создаем (например, «Отклонять запросы API, превышающие лимит, с HTTP 429»).
* **Нецель:** То, что выходит за рамки данной спецификации и будет делаться позже (например, «Построение панели биллинга»).

## 2. Критерии приемки (Acceptance Criteria)
Условия, по которым будет проверяться финальный код (будут использоваться для написания тестов):
1. [ ] Условие 1 (например: Неаутентифицированные запросы получают 401 до проверки лимита).
2. [ ] Условие 2.

## 3. Открытые вопросы (Open Questions)
Список нерешенных архитектурных или продуктовых вопросов. Обязательно разрешите их перед написанием кода.
* **Вопрос 1:** Описание проблемы
* **Вопрос 2:** Описание проблемы

## 4. Архитектура и затронутые модули (Architecture)
Какие микросервисы, таблицы или пакеты изменятся.
* **Модуль/Сервис A:** [Краткое описание изменений]
* **Модуль/Сервис B:** [Краткое описание интеграции]

## 5. Модель данных и контракты API (Data & API Contracts)
* **Схема данных:** Новые таблицы, поля, требования к индексам.
* **Контракты (для HTTP API или событий):**
  * Метод: `POST /api/v1/resource`
  * Payload запроса: `{"field": "type"}`
  * Payload ответа: `{"status": "success"}`
  * Коды ошибок: `401, 404, 429`

## 6. Безопасность и Наблюдаемость (Security & Observability)
* **Ограничения безопасности:** Права доступа (RBAC), шифрование, защита от утечек.
* **Логирование/Метрики:** Ключевые события, которые нужно логировать, алерты для мониторинга.

## 7. Стратегия тестирования (Testing Strategy)
* **Unit-тесты:** Какую критическую логику покрываем.
* **Интеграционные тесты:** Основные сценарии взаимодействия.
* **Ручное ревью:** Что именно должен проверить человек (UX, крайние случаи ИИ).

## 8. Заметки по миграции (Migrations)
* Шаги по развертыванию на продакшене.
* Пошаговая инструкция для отката (rollback), если что-то пойдет не так.
